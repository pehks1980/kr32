# KR32 Memory Map

The active scheduler uses dynamically allocated page tables and physical
pages. Every task sees the same user data and stack virtual addresses, while
those addresses resolve to different physical pages.

User virtual layout
===================

0x00005000  USER_BASE

0x00006000  USER_DATA_VA
            One read/write user page per task.

0x00009000  Current idle-task executable VA.
0x00019000  Current task-A executable VA.
0x0001A000  Current task-B executable VA.
            Executable pages retain their assembled VAs for now and are
            mapped only into the owning task.

0x0003F000  USER_STACK_VA
            One read/write user stack page per task.

0x00040000  USER_STACK_TOP
            Initial SP for every task. The stack grows down into 0x3F000.
            This address is also one byte above USER_LIMIT.

User pointers are valid only in:

    USER_BASE <= address <= USER_LIMIT (0x0003FFFF)

Every page in a syscall buffer must additionally be present, user-accessible,
and have the requested read/write permission in the current task's page table.

Dynamic kernel physical pool
============================

0x00040000  PAGE_ALLOC_BASE
0x000C0000  PAGE_ALLOC_END (exclusive)

The allocator manages 128 4-KiB pages. The complete pool is identity-mapped
in every address space with supervisor-only global permissions:

    VA physical_address -> PA physical_address [P R W X G]

The pool currently contains dynamically allocated:

- Page tables
- Kernel stacks
- Per-task file descriptor tables
- Per-task kernel read/write buffers
- User data backing pages
- User stack backing pages

Because these mappings do not have PTE_U, user mode cannot access the pool.
Keeping the pool identity-mapped lets kernel code use the physical addresses
stored in task structures directly.

Per-task mappings
=================

Each task owns a dynamically allocated PTBR. A typical task maps:

    USER_DATA_VA  -> task-specific allocated PA [P R W U]
    USER_STACK_VA -> task-specific allocated PA [P R W U]
    entry page    -> current assembled executable PA [P R X U]

The idle task runs in supervisor mode, so its entry page is supervisor-only.

Common kernel mappings
======================

Static kernel code/data pages and MMIO pages remain identity-mapped,
supervisor-only, and global. The dynamic allocator pool is mapped the same
way. User executable pages are not common/global mappings.

Boot sequence
=============

1. Clear the physical-page allocator bitmap.
2. `init_scheduler` creates tasks and their page tables dynamically.
3. Load task zero's PTBR and kernel trapframe SP.
4. Enable the MMU.
5. Enter `trap_restore`.

The old static `TASK0_PTBR`/`TASK1_PTBR`/`TASK2_PTBR` areas and
`init_scheduler1` remain only as legacy reference code; the active boot path
does not use them.
